# Munim.ai — User Flow & Problem Statement

---

## The Problem We're Solving

India has **63 million MSMEs**. Most of them buy goods from multiple suppliers every month. Under GST, every purchase invoice entitles the buyer to claim back the tax they paid — this is called **Input Tax Credit (ITC)**.

**The reality for a kirana store owner, textile trader, or hardware dealer:**

- They get 30–80 paper invoices every month from different suppliers
- They don't know which invoices are GST-compliant and which aren't
- They don't know if their supplier actually filed their tax return (if not, the ITC is blocked)
- They don't know if the HSN code on the invoice is wrong (wrong code = wrong tax rate = blocked ITC)
- They hand everything to a CA at the end of the month, who charges ₹2,000–5,000 just to process it
- **Average trader loses ₹8,000–25,000/month in ITC they never claimed or that got blocked**

Every existing tool — ClearTax, Vyapar, Tally, GSTHero — is designed **for CAs**. A trader in Lucknow or Surat can't use them. Too complex, too English, no guidance.

**Munim.ai flips this.** The trader is the customer. The munim (your AI bookkeeper) runs 24/7 on WhatsApp, speaks Hindi, and tells the trader in plain language: *"Is invoice mein ₹1,200 ka ITC block hai — supplier ne return file nahi kiya. Unhe call karo."*

**What happens in the backend on every invoice:**
1. Image downloaded from WhatsApp
2. Gemini Vision extracts: invoice number, date, supplier GSTIN, buyer GSTIN, all line items, HSN codes, tax amounts
3. GSTIN verified against government API (cached in Redis)
4. All HSN codes validated against 12,000+ code database + pgvector semantic search
5. Invoice matched against GSTR-2B records (3-pass fuzzy matching)
6. ITC verdict computed by deterministic rules engine (GST Act Sec 16 + 17(5))
7. Fraud scored on 5 signals: GSTIN age, Benford's Law, sequential invoice numbers, geographic mismatch, velocity anomaly
8. Hindi diagnosis generated by Gemini
9. Invoice stored in Supabase with all fields, line items, fraud signals, match status
10. Supplier auto-created and linked (builds compliance graph)

## What the System Checks on Every Invoice

| Check | What It Looks For | Outcome |
|-------|------------------|---------|
| **GSTIN Validation** | Is supplier registered? Active? Not cancelled? | INELIGIBLE if invalid |
| **Section 17(5) Check** | Motor vehicle, beauty services, personal consumption, outdoor catering? | Hard block — ITC never allowed |
| **Section 16(2) Check** | Valid tax invoice? All mandatory fields present? | FIXABLE if missing fields |
| **Section 16(4) Check** | Invoice older than 18 months? | INELIGIBLE — time limit expired |
| **HSN Validation** | Is HSN code in the official 12,000+ code database? Correct GST rate? | FIXABLE if rate mismatch |
| **GSTR-2B Match** | Did supplier actually file this invoice in their GSTR-1? | AT_RISK if not found |
| **Fraud Scoring** | 5 statistical signals (see below) | FRAUD_FLAGGED if score ≥ 70 |

### Fraud Signals (5 of them)
| Signal | What It Detects |
|--------|----------------|
| **GSTIN Age** | New GSTIN (< 6 months old) issuing high-value invoices |
| **Benford's Law** | Unnatural digit distribution in invoice amounts — statistical signature of fabricated numbers |
| **Sequential Invoice Numbers** | Consecutive invoice numbers from same supplier — common in fake invoice mills |
| **Geographic Mismatch** | Supplier state ≠ buyer state but no IGST charged |
| **Velocity Anomaly** | Current invoice 5x+ higher than this supplier's historical average |
